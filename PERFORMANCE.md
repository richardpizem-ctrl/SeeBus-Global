# PERFORMANCE – SeeBus‑Global

This document describes the performance characteristics, optimizations, and recommended tuning strategies for the SeeBus‑Global backend.  
The goal is to ensure fast ETA computation, stable GTFS‑RT processing, and predictable behavior under load.

---------------------------------------

## 1. Performance Goals

The system is designed to meet the following targets:

- GTFS‑RT processing: < 150 ms per update cycle  
- ETA computation: < 10 ms per request  
- API response time: < 50 ms under normal load  
- Memory footprint: < 200 MB for medium‑sized cities  
- Cache refresh: < 30 ms  

These values depend on dataset size and hardware.

---------------------------------------

## 2. Static GTFS Loading

Static GTFS files are loaded once at startup:

- stops.txt → Stop objects  
- routes.txt → Route objects  
- trips.txt → Trip objects  
- stop_times.txt → stop sequences  

Optimizations:

- Pre‑indexed maps for O(1) lookup  
- Pre‑computed stop sequences  
- Pre‑computed route → trips mapping  
- Lazy loading of optional fields  

---------------------------------------

## 3. GTFS‑RT Processing

GTFS‑RT feeds (TripUpdates, VehiclePositions) are processed every N seconds.

Optimizations:

- Only changed trips are updated  
- Old entries are expired using TTL  
- Vehicle positions stored in a compact structure  
- Delay values normalized immediately  
- Merging static + RT data done in O(1) per trip  

Typical processing time: 50–120 ms depending on feed size.

---------------------------------------

## 4. Caching Strategy

The backend uses multiple in‑memory caches:

stops_cache: map<stop_id, Stop>  
routes_cache: map<route_id, Route>  
trips_cache: map<trip_id, Trip>  
vehicle_positions_cache: map<trip_id, VehiclePosition>  
trip_updates_cache: map<trip_id, TripUpdate>  

Cache TTL is controlled by:

CACHE_TTL_SECONDS

Benefits:

- Eliminates repeated disk reads  
- Reduces GTFS‑RT merge cost  
- Ensures predictable API latency  

---------------------------------------

## 5. ETA Computation Performance

ETA calculation is optimized for speed:

- O(1) lookup for next stop  
- O(1) lookup for delay  
- O(1) lookup for vehicle position  
- Lightweight arithmetic only  

ETAResult is computed in under 10 ms even on low‑end hardware.

---------------------------------------

## 6. Event Engine Performance

Events (ARRIVAL_IMMINENT, AT_STOP, etc.) are generated by comparing:

- current_stop  
- next_stop  
- ETA  
- delay  
- vehicle position  

Optimizations:

- No heavy loops  
- No historical storage  
- Stateless evaluation per request  

Event generation cost: < 5 ms.

---------------------------------------

## 7. API Performance

Endpoints are designed to be lightweight:

- No database queries  
- No disk access  
- No heavy JSON parsing  
- No blocking operations  

Typical response times:

- /eta → 5–20 ms  
- /journey → 10–30 ms  
- /events → 5–15 ms  

---------------------------------------

## 8. Scaling Recommendations

### Vertical scaling
- Increase CPU for faster GTFS‑RT processing  
- Increase RAM for larger GTFS datasets  

### Horizontal scaling
- Run multiple backend instances  
- Use a load balancer (Nginx, HAProxy)  
- Use sticky sessions only if needed (rare)  

### Cache tuning
- Lower TTL for more real‑time accuracy  
- Higher TTL for lower CPU usage  

---------------------------------------

## 9. Stress Test Guidelines

Recommended tests:

- 1000 ETA requests per second  
- 10× GTFS‑RT feed size  
- 24‑hour continuous run  
- Randomized journey creation  

Metrics to monitor:

- CPU usage  
- Memory usage  
- Cache hit ratio  
- GTFS‑RT processing time  
- API latency  

---------------------------------------

## 10. Known Bottlenecks

- Extremely large GTFS datasets (> 200k stops)  
- Very frequent GTFS‑RT updates (< 1 second)  
- Slow or unstable GTFS‑RT provider  
- Excessive logging in debug mode  

---------------------------------------

## 11. Optimization Tips

- Use uv instead of pip for faster startup  
- Disable debug logs in production  
- Keep GTFS static files on SSD  
- Use systemd or PM2 for stable restarts  
- Increase TTL if CPU usage is high
